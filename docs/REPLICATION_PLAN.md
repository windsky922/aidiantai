# Claudio AI 电台复现方案

## 1. 对原项目的理解

作者给出的内容其实分成两个层次。

第一层是公开页面 `https://mmguo.dev/claudio-fm/`。这是一个静态节目播放器，用来展示最终体验。页面里包含 AI DJ 名称、头像、开场音频、歌曲 preview、字幕时间轴、波形动画和播放控制。它没有公开完整后端，也没有实时调用 AI 大脑、网易云或 Fish Audio。

第二层是视频里展示的本地系统架构。这个系统叫 Claudio，原作者示意里使用 Claude / Claude Code 作为“大脑”。本复现项目的目标实现改为使用 Codex：由本地 Node.js 服务把用户资料、歌单、天气、日程、历史播放和工具结果组装成上下文，再交给 Codex 决定此刻该说什么、播什么、为什么播。

因此复现时不能直接从“完整自动电台”开始。更稳妥的路线是先复刻静态播放器体验，再逐步把静态内容替换成 AI 生成内容。

## 2. 产品定位

这个项目不是普通音乐播放器，也不是简单的 AI 歌单生成器。它应该是一个个人上下文电台。

普通播放器的核心问题是“播放哪首歌”。Claudio 类型项目的核心问题是：

- 现在是什么时间？
- 用户今天处在什么状态？
- 用户过去多年喜欢过什么音乐？
- 当前天气、日程、场景适合什么声音？
- AI DJ 应该用什么语气解释这次播放？
- 这段电台能不能像一个真正的节目，而不是随机推荐？

最终体验应该接近：

> 打开页面后，一个私人 AI DJ 根据我的时间、天气、歌单和习惯，说一段自然的开场白，然后播放一首合适的歌，并解释为什么现在适合它。

## 3. 作者原页面拆解

原页面可以拆成这些模块：

- `stage`：全屏背景和流体渐变。
- `card`：移动端播放器卡片。
- `header`：Claudio 名称、头像、状态、当前时间。
- `waveCanvas`：音频波形可视化。
- `meta`：节目名和歌曲名。
- `mini-player`：歌曲 preview 小播放器。
- `transcript`：按时间滚动的转录字幕。
- `scrubber`：节目进度条。
- `tweak-panel`：调试视觉参数。

页面数据特点：

- `RAW_TURNS` 是写死的转录时间轴。
- `INTRO_SRC` 是本地 AI DJ 开场音频。
- `SONG.previewUrl` 使用 Apple Music / iTunes 30 秒预览音频。
- Web Audio 只负责分析背景音乐频谱并驱动波形。
- 没有真实后端 API 请求。

这个页面适合作为第一阶段的前端目标。

## 4. 视频架构拆解

根据视频截图，完整系统可以还原为四层。

### 第一层：用户上下文

用户资料文件：

- `taste.md`：音乐品味、长期偏好、讨厌什么。
- `routines.md`：作息和日常节奏。
- `playlists.json`：歌单、歌曲、歌手、年份、来源。
- `mood-rules.md`：天气、时间、情绪和音乐匹配规则。

外部上下文：

- 天气
- 日历
- 当前时间
- 最近播放
- 最近对话
- 用户反馈

### 第二层：本地大脑

本地 Node.js 服务负责调度：

- `router.js`：识别用户意图，决定走音乐、聊天、计划还是工具调用。
- `context.js`：把用户资料、环境、历史和工具结果拼成 prompt。
- `codex.js`：调用 Codex，要求输出结构化 JSON。
- `scheduler.js`：按时间触发早间、午后、深夜等节目。
- `tts.js`：调用 Fish Audio 生成 DJ 语音，并做缓存。
- `state.db`：保存播放、对话、计划和反馈。

### 第三层：工具 API

- Codex：节目策划、推荐逻辑、DJ 文案、解释原因。
- 网易云音乐 API：搜索歌曲、歌曲链接、歌词、推荐。
- Fish Audio：把 DJ 文案合成为语音。
- 飞书 API：读取日程。
- OpenWeather：读取天气。
- UPnP：推送到家庭音响。

### 第四层：PWA 播放器

至少包含：

- `Player`：当前节目、当前歌曲、DJ 字幕、播放控制。
- `Profile`：音乐画像、偏好、年份分布。
- `Settings`：API key、音色、网易云配置、缓存设置。

前后端通信：

- `GET /api/now`：当前播放状态。
- `GET /api/taste`：用户音乐画像。
- `GET /api/plan/today`：今日节目计划。
- `POST /api/episode`：生成一期节目。
- `POST /api/feedback`：记录喜欢、不喜欢、跳过。
- `WS /stream`：推送 now-playing、字幕、状态变化。

## 5. 推荐技术栈

### 最小可行版本技术栈

- 前端：Vite + React + TypeScript
- 样式：CSS Modules 或 Tailwind CSS
- 后端：Node.js + Fastify
- 数据库：SQLite
- 音频：原生 Audio + Web Audio API
- AI：Codex
- TTS：Fish Audio

### 后续增强

- 向量检索：pgvector、Chroma 或 LanceDB
- 歌曲分析：Essentia、Spotify Audio Features 替代源、Librosa
- 队列管理：BullMQ
- 桌面服务：Tauri 或 Electron
- 部署：本地优先，远程只放不含版权音乐的展示页

## 6. 分阶段实施

### 阶段 0：仓库和文档

目标：

- 建立项目文档。
- 明确复现路线。
- 建立任务日志和 GitHub 同步规范。

产出：

- `README.md`
- `docs/REPLICATION_PLAN.md`
- `docs/TASK_LOG.md`

### 阶段 1：静态播放器演示

目标：

- 复现作者公开页面的核心体验。
- 不依赖后端，不接 AI。

功能：

- 移动端电台卡片 UI。
- Claudio 名称、头像和状态。
- 播放 / 暂停。
- 一段 DJ 音频。
- 一段歌曲预览音频或本地音频。
- 同步字幕。
- 波形动画。
- 进度条。

验收标准：

- `npm run dev` 后可在 localhost 访问。
- 点击播放后，DJ 音频和背景音乐同步播放。
- 字幕按时间高亮和滚动。
- 移动端视口不溢出。

### 阶段 2：节目数据模型

目标：

- 把写死的页面内容改造成数据驱动。

新增结构：

```json
{
  "episodeTitle": "mmguo's Pilot Episode",
  "host": "Claudio",
  "introAudio": "/audio/intro.mp3",
  "song": {
    "title": "If",
    "artist": "Bread",
    "previewUrl": "https://..."
  },
  "turns": [
    {
      "who": "Claudio",
      "start": 0,
      "text": "This is Claudio."
    }
  ]
}
```

验收标准：

- 前端只读取 JSON 数据渲染节目。
- 更换 JSON 就能更换一期节目。

### 阶段 3：本地服务器

目标：

- 加入 Node.js 服务，前端从后端读取节目数据。

接口：

- `GET /api/now`
- `GET /api/episodes/:id`
- `POST /api/episode`
- `WS /stream`

验收标准：

- 页面不再写死节目内容。
- 后端可以返回当前节目和播放状态。

### 阶段 4：歌单导入和用户上下文

目标：

- 建立个人资料文件和歌单结构。

文件：

- `data/taste.md`
- `data/routines.md`
- `data/mood-rules.md`
- `data/playlists.json`

验收标准：

- 后端能读取这些文件。
- 后端能根据时间和规则筛选候选歌曲。

### 阶段 5：Codex 节目策划

目标：

- 让 Codex 根据上下文生成一期节目。

Codex 输出：

```json
{
  "episodeTitle": "周日晚上的旧耳机",
  "djSay": "现在适合把节奏慢下来。",
  "songCandidates": [
    {
      "title": "If",
      "artist": "Bread",
      "reason": "低能量、温柔、适合深夜。"
    }
  ],
  "segue": "下一首会更安静一点。"
}
```

验收标准：

- 输入相同上下文时结果稳定。
- 输出 JSON 可解析。
- 失败时有兜底输出。

### 阶段 6：TTS 和音频缓存

目标：

- 用 Fish Audio 或其他 TTS 把 DJ 文案转成音频。

缓存：

- `cache/tts/{hash}.mp3`

验收标准：

- 相同文本不重复生成。
- TTS 失败时前端仍能显示字幕。
- 可配置主播音色。

### 阶段 7：音乐 API

目标：

- 接入网易云音乐 API 或替代音源。

能力：

- 搜歌
- 获取歌词
- 获取 preview 或可播放链接
- 推荐相似歌曲

注意：

- 公开部署不能直接分发版权音乐。
- 公开演示优先使用官方预览音频、本地用户文件或无版权音乐。

### 阶段 8：智能调度和长期记忆

目标：

- 让电台主动运转。

触发：

- 早上生成今日计划。
- 工作时播放低干扰音乐。
- 下雨时切换安静频道。
- 日程前后调整能量。
- 用户跳过时记录反馈。

验收标准：

- `state.db` 记录历史。
- 反馈会影响下一次推荐。
- 定时任务可开关。

## 7. 可以优化的位置

### 体验优化

1. 从“单期节目”扩展为“频道系统”

   不只生成一首歌，而是生成多个频道：

   - 深夜私人电台
   - 工作流频道
   - 下雨天频道
   - 十四年回忆频道
   - 通勤恢复频道

2. 加入音乐时间线

   把十四年歌单做成年份地图，展示每年高频歌手、情绪和关键词。

3. 做真实反馈闭环

   播放器提供：

   - 喜欢
   - 不喜欢
   - 太吵
   - 太困
   - 换个年代
   - 多说一点
   - 少说一点

4. 分离“DJ 讲话”和“音乐播放”

   用户可以调节：

   - DJ 说话频率
   - DJ 风格
   - 音乐音量
   - 背景音乐 ducking 强度

### 技术优化

1. Prompt 版本化

   把 prompt 放在 `prompts/`，每次改动可追踪。

2. 结构化输出校验

   使用 Zod 校验 Codex 输出，避免坏 JSON 破坏播放器。

3. TTS 成本控制

   对文本 hash，缓存音频。短句合并生成，减少请求次数。

4. 音乐候选先规则筛选，再交给 Codex

   不要让 Codex 从几千首歌里盲选。先用规则或 embedding 选出 20 首候选，再让 Codex 排序和解释。

5. 后端事件流

   使用 WebSocket 或 Server-Sent Events 推送状态，避免前端轮询。

6. 版权边界

   公开页面只使用：

   - 官方 preview
   - 用户本地文件
   - 无版权音乐
   - 用户自行授权的音频

### 产品优化

1. 从“播放器”升级为“个人声音主页”

   作者原页面已经有这个方向：它不只是播歌，还像一个可长期更新的个人电台入口。

2. 支持节目导出

   把一次 AI 电台导出为：

   - 分享页面
   - 音频文件
   - 视频素材
   - 播放清单

3. 支持多 DJ 人格

   例如：

   - 克制型
   - 深夜型
   - 日记型
   - 资讯型
   - 朋友型

4. 支持“为什么播这首”

   这是 AI 电台和普通推荐系统的核心差异。每首歌都应该有解释。

## 8. 当前最推荐的下一步

下一步应实现阶段 1：静态播放器 Demo。

原因：

- 最快获得可见成果。
- 可以验证 UI、音频同步、字幕、波形这些核心体验。
- 不会被 Codex 集成、TTS、网易云接口和版权问题拖住。
- 后续后端只需要替换数据来源，不推翻前端。

阶段 1 完成后，再把节目内容从写死数据升级成 JSON，并继续接入本地服务。
